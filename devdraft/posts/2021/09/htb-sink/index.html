Searchsploit has no exploit on gunicorn; cvedetails and Mitre do a very poor job to identify a vulnerability while Snyk.io find an http request smuggling vulnerability which affects gunicorn-20.0.0 and documented. It is confirmed by the 2019 changelog

There is a bug where haproxy and gunicorn do not handle in the same way discrepancies between Content-Length and Trans-Encoding headers resuting in htt request smuggling. The gunicorn team discuss how to fix the Trans-Encoding in this issue while this blog explains the exploitation of haproxy

Http smuggling background

This video: HTTP Request smuggling explain | HAProxy HTTP Request smuggling Vulnerability | CVE-2019-18277 POC resumes how to perform the attack.

When sending to HAProxy \x0b (vertical tab) or \x0c (form feed) in Transfer-Encoding header:

1
2
3

Transfer-Encoding:\x0bchunked
[...]
0

it results in HAProxy sending to the backend the following, with chunked being on the next line, and therefore the Transfer-Encoding may be ignore by backend and Content-Length will take priority and be used instead.

1
2

Transfer-Encoding:
	chunked
