How I built this privacy friendly blog

privacy github disqus gohugo

Why I built this blog

Social Media dominant ideology is corrupting people thought

In an increasingly manicheist world loosing its sense of shades and scales of Tolerance towards extremes sides, I found difficult to publicly express my view of facts and events that may contradicts dominant ideology. I was looking for a way to post my opinions and thoughts online while avoiding the new norm of social media pressure that systematically judges you according to some political prism or psychological bias.

While social network is the right place to reach the mass, it is certainly not the forum were you can speak freely and openly without consequences, and I felt my wording will be influenced by my readers and not honestly reflects what is on my mind. Thinking a bit further I have realized that my readers may want to comment as well, and I was wondering how I could offer them this feature with the same level of privacy I am advocating in this post.

This is the limit of freedom of speech when you apply self-censorship to avoid hypothetical consequences that may or may never occure, such as loosing your job, contracts or opportunities because some folks may not like what you have to say.

So well I have chosen to keep my ambition small with a limited reach and without associating my name in a too obvious manner to whoever may want to use what I have posted online against me. I bet that those who find value in what I post will share and promote its content or follow some of the present guidelines in this article to safeguard their own privacy, and I trust that those who know me will respect that and avoid to disclose my identity.

Anonymity vs Pseudonymity vs Privacy

Purpose in here has never been and will never be anonymity: any motivated actors or government can easily track back to my personal identity. Pseudonymity is more appropriate to describe in which way you can reach a much better level of privacy that prevents mass surveillance of citizen’s opinion as well as retorsion in your company.

How I built this blog

a private email address to register github pages

  1. I have first created an anonymous email address for the solely purpose to register a github account without sharing my personal details or email. Any privacy-friendly email provider such as protonvpn or tutanota will be fine, as long as they don’t track you. I won’t use this email or email alias for any other purpose.

  2. As a first approach and before investing in domain name and hosting, we can host our website for free on github pages. The website will be accessible with username.github.io, so it is better not to use an overly complicated or long username, short is better. Should I mention that putting your real name as username, defeats the purpose of this article?

A statically generated website not to be hacked

What is common with all CMS (Content management system) I know such as wordpress, drupal, joomla … is how frequently hackers exploits zero-days or disclosed vulnerabilities to compromise those websites. Whether the backend is PHP, ASP.Net, Java, Nodejs, … does not matter as much as the perspective for a hacker to takeover the website backend to remotely execute code. Another major issues are backend and software updates (operating system, CMS …) which may not be performed timely, or may break the website between major upgrades. I personnally don’t want to spend time patching things of fixing things after patching.

To avoid the above issues, I made the decision to go for a statically generated website with Hugo As there is no server-side code, it is closed to impossible to takeover the backend and a more probable attack vector is to compromise instead the github credentials, tokens or ssh keys … which are not accessible from the website itself, but from my laptop which therefore becomes the most likely attack vector. While, most of the security of the blog is managed by github directly, there is a set of good practices that I generally recommend, to avoid being the weakest link:

  • use a password manager to vault your email and github credentials, with strong password encryption
  • secure your ssh keys with a passphrase
  • use a different ssh keys per repositories
  • github token should have the minimum permissions as possible
  • never reuse an existing github token for a different purpose, create a new token instead

A deployment strategy to mitigate your real identity leak

Manual is always error-prone and may result in leaking my identity by mistake. I have therefore decided to automate the build and deployment of the hugo static website into GitHub Pages via GitHub Actions. Creating a Hugo website and setting up a GitHub pages is out of scope for this article.

There is mostly two strategies to deal with deployment:

  • use a single repository with two branches: main branch for source code and gh-pages for published website
  • use separate repositories: the source code is in a private repository and receive manual commits, github-actions auto-commit static pages into the gh-pages repo

The separate repositories approach works better on a pure SecOps perspective, but it is not sufficient as it could be possible to inadvertently leak information on the articles content itself. Therefore I have to consider the review process prior github-actions automatically build and deploy content to gh-pages.

A comment engine which is respectful of your privacy

When it comes with blogging, we want to interact with people and receive feedback. This interaction is priceless to confront ideas and build better ones. While there is plethor of engines to offer incredibly friendly comment features on a static blog, I found the privacy price-tag incredibly high, and most of those solutions were quite expensive.

Again whether cheap or expensive, money trail implies easy back-tracking to people real identity

Without a backend, I need to use javascript tokens and a third-party engine such as infamous disqus, and I can’t keep to question the security of such tokens: it could be used by someone malicious to flood me with thousands of fake comments. Another option could be facebook comments which requires user to be tracked by facebook instead of disqus.

Such third-parties tracking my audience and my website traffic totally goes against my principles, whichever the reason (targeted ads…). I found instead that anybody can create an anonymous or pseudonymized github account like the gh-pages account running this website, and this will allow them to comment or contribute to my privacy project without a need to disclose their real identity nor register an external services.
